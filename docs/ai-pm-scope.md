# AI で PJ 管理どこまでできるか（スコープと限界）

「AI PMO」と言ったとき顧客から必ず聞かれる "結局どこまで AI ができるの？" に答える整理。
[[pmo-roles-map]] が業務の網羅マップ、本ノートは**システム開発 PJ で実用できる粒度**まで掘る。

> 骨格のみ。Keita レビュー後、実証データや事例で肉付けする。

---

## 1. 「できる」「できる が条件付き」「できない」の3層

### ◎ 完全に AI に任せられる（人間は最終確認のみ）

| 業務 | 具体的なやれること |
|----|-------|
| 議事録作成 | 録音→文字起こし→構造化要約→決定事項抽出 |
| 週次/日次レポート生成 | Jira/Backlog からデータ取得→分析→所定フォーマット出力 |
| 遅延タスク検出 | WBS/期限を監視し、閾値超過時にアラート |
| ドキュメント検索・要約 | 過去ドキュメントから関連箇所を抽出して要約 |
| FAQ 一次回答 | 過去 Q&A・ドキュメントから一次回答案を提示 |

### ○ AI が下書きを作り人間が判断（半自動）

| 業務 | 限界・条件 |
|----|------|
| リスク識別 | 過去事例 + 現状データから候補抽出。重大度判断は人間 |
| 課題分類・優先順位付け | 自動分類は可能。優先順位は事業文脈が必要で人間判断 |
| 工数予測 | 過去データがあれば予測可。組織特有のクセは要チューニング |
| ステークホルダーレポート | 下書き可。トーン・政治的配慮は人間調整 |
| ドキュメントレビュー（整合性） | 矛盾検出は得意。仕様の妥当性判断は人間 |

### △ AI は補助（人間主導）

| 業務 | AI の貢献範囲 |
|----|--------|
| 要員アサイン | スキル・稼働マッチング候補提示まで。最終判断は人間 |
| 顧客交渉準備 | 過去交渉の論点整理・想定問答作成 |
| ベンダー選定 | 候補比較表作成。価値判断は人間 |
| プロジェクト標準策定 | 既存標準のドラフト・ベンチマーク提示 |

### × 現状やるべきでない

| 業務 | 理由 |
|----|----|
| 契約締結判断 | 法的責任 |
| 役員報告での最終判断 | 政治・責任問題 |
| 人事評価 | 倫理・法令（AI 法、労働法） |
| インシデント時の対外公表判断 | レピュテーション責任 |
| 監査での回答 | 法的責任 |

## 2. システム開発 PJ 固有のスコープ

システム開発という具体ドメインで何が「特に効く」か。

### 効果が大きい領域

- **コードレビュー連携**: GitHub PR/コミットの監視と進捗反映
- **障害分析・原因仮説生成**: ログ・Slack スレッドからの一次分析
- **テスト計画レビュー**: テストケース網羅性チェック
- **リリース管理**: 変更内容の影響範囲推定・リリースノート自動生成
- **要件・設計書の整合性チェック**: ドキュメント横断レビュー

### 効果が限定的な領域

- **見積もり** — 組織のクセが強く、データが揃わないと精度出ない
- **顧客との折衝** — 信頼関係依存
- **チームのモチベ管理** — 人間の領域

## 3. 「どこまで」の天井を上げる方針

短期（今すぐ）:
- 既存ツール連携を厚くして、データソースを広げる
- 出力フォーマットを顧客の既存テンプレに合わせる

中期（半年〜1年）:
- RAG / メモリ機構を強化して、組織固有の暗黙知を吸収
- マルチエージェント協調（[[pmo-roles-map]] の構成案）

長期:
- 過去案件データの蓄積で予測精度を上げる
- AI が「次に何をすべきか」を自律的に提案

---

## 4. 顧客説明での誤解を避ける鉄則

1. **「AI が PM の代わりをする」とは絶対に言わない** — AI は PMO 補佐
2. **数字を見せて期待値を揃える** — 「議事録工数 80% 削減」など実測値で
3. **失敗事例も共有** — ハルシネ・誤判断のリスクは隠さない
4. **段階導入を前提に** — 一気に全領域は無理。優先業務から

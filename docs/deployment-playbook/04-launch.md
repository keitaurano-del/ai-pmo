# Phase 4: ローンチ（1週間）

## 目的

AI PMO エージェント群を **対象 PJ で初めて稼働** させ、初期成果物をクライアントとレビューする。
ここがクライアントの「驚き」を生む最大の山場。

## 期間

- 標準: 5営業日（1週間）

## 担当

- **AI PMO エンジニア 1名**（生成・レビュー）
- **Sr. コンサル 1名**（クライアント対面）
- クライアント側: PM + PMO 担当（毎日 1h レビュー枠を確保）

## アクティビティ

### Day 1: 初期 AI 生成セッション

過去 1ヶ月のデータを基に、AI が **PJ 全成果物の初稿** を一気に生成：

```bash
npm run agent:run -- --client spring-foods --agent reporter --mode initial
npm run agent:run -- --client spring-foods --agent scribe --mode backfill
npm run agent:run -- --client spring-foods --agent triager --mode initial
npm run agent:run -- --client spring-foods --agent watcher --mode initial
```

生成される成果物：
- WBS（既存タスクから階層構造を AI が推定）
- 過去 1ヶ月の議事録（Zoom 録音から）
- 課題管理表（Backlog チケットから AI が分類）
- リスク台帳（議事録 + Slack 投稿から AI が抽出）
- 直近週次レポート（バックフィル分）
- 意思決定ログ（議事録から抽出）

→ 約 50〜100 ファイルが数分で生成される。

### Day 2: 内部レビュー会（コンサル + エンジニア）

- 生成物の **明らかな誤り** を修正
- カテゴリ揺れ・表記揺れを統一
- 機密情報の漏れ込みチェック
- 「クライアントに見せる前」のクリーニング

**目安**: 生成物の 30% に修正が入る（許容範囲）。50% を超えるならエージェント設定の見直しが必要。

### Day 3: クライアント側 PMO レビュー会（半日）

クライアント PMO 担当 + PM を集めて、生成物を一緒に見る：

- ビューワー（`https://ai-pmo-<slug>.onrender.com`）を画面共有
- 各成果物を順に見せて、率直なフィードバックを集める
- 修正要望は GitHub Issue（or Linear）に起票

**典型的なフィードバック**：
- 「この用語、社内では別の言い方をしている」→ glossary.md 追加
- 「この課題は重要なのに Low になっている」→ triager のルール調整
- 「報告書のフォーマットを経営層向けにもう少し簡潔に」→ テンプレ修正

### Day 4: 修正反映 + 再生成

- フィードバックを反映してエージェント設定・テンプレを更新
- 再度 AI 生成を実行
- クライアント PMO 担当に「修正版」を共有

### Day 5: キックオフ会議

経営スポンサー + PMO 室長 + PM + AI PMO チーム全員を集めた **正式キックオフ**：

- AI PMO の役割説明（再確認）
- 生成物デモ（ビューワーで）
- 運用ルール合意（毎週月曜の週次定例で AI PMO 進捗を確認 等）
- 緊急時のロールバック手順確認
- KPI ベースラインの確認

→ 正式に **Phase 5: 運用** へ移行。

## クライアントが驚くポイント（営業的に重要）

1. **過去 1ヶ月の議事録を全部もう一度作り直してくれた** — Zoom 録音から自動生成
2. **未起票の課題を AI が拾ってくれた** — 議事録の「やる」発言を自動抽出
3. **これまで月次だった経営報告が、PoC 開始から1週で出てきた** — リアルタイム可視化
4. **重複している Backlog チケットを AI が指摘** — トリアージャーの仕事
5. **既存 PMO ジュニアでもベテラン並の品質のレポートが書ける** — AI が下書き

これらを **Day 3 のレビュー会** で必ず見せ、Sr. コンサルが意図的に強調する。

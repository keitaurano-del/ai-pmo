# 運営ガイド — AI PMO の使い方（PM / PMO 担当 向け）

> Phase 5 運用フェーズに入った PJ で、**PM / PMO 担当者がどう AI PMO を日常業務に組み込むか** の具体的な手順書。

## 朝一でやること（毎営業日、5〜10 分）

### 1. AI PMO ビューワーのダッシュボードを開く

URL: `https://ai-pmo-<client>.onrender.com/<case>`

確認順：
1. **KPI スナップショット** — 全体進捗が計画から大きく外れていないか
2. **High 課題** — 新規発生・期限間近の案件はないか
3. **重点監視リスク** — 昨日からスコア変化があった項目
4. **最新ドキュメント** — 前日に新しく生成された議事録・報告書

→ ここで「今日対応すべき緊急 1〜2 件」を把握する。所要 3 分。

### 2. Slack #ai-pmo-alerts チャンネルを確認

watcher-agent が前夜 06:00 に走らせた検知結果が朝の段階で投稿されている。

- 🚨 マークの付いた High アラート → 即対応
- ⚠️ マークの付いた Mid → 午前中に確認

### 3. 当日の会議予定を確認

- 自分が出る会議が複数ある場合、scribe-agent が録音から議事録を作るので **録音 ON を忘れない**
- AI PMO 担当が参加できない会議は、議事録のレビュー担当を事前にアサイン

## 会議運営（毎回、最大 15 分の追加負担）

### 会議前

- Zoom or Teams のレコーディングを **必ず ON にする**
- 字幕（クローズドキャプション）も ON にしておくと scribe-agent の精度が上がる
- アジェンダは前日までに Slack の会議スレッドに投稿（reporter-agent が次回参考にする）

### 会議中

- 通常通り進行
- 「決定」「やる」「期限」などのキーワードは **意識的に発声**（AI の抽出精度向上のため）
- センシティブな内容（給与・人事評価・他社の機密）に触れる時は、AI が拾わないよう「これは記録外」と前置きする

### 会議終了 1 時間後

scribe-agent が議事録を `client-env/04-meetings/YYYY-MM-DD_xxx.md` に保存し、Slack で通知。

レビュー手順（10 分）：

1. 議事録ファイルを開く
2. **決定事項** の抽出漏れ・誤抽出をチェック
3. **ToDo** が正しく担当者・期限付きで抽出されているか確認
4. 違っている箇所を直接編集
5. ファイル末尾の `Reviewed by: <自分の名前>` を記入
6. Slack スレッドに「議事録レビュー完了」と返信

→ 議事録の確定リードタイムが**当日中**になる。

## 週次ルーチン（月曜の朝、30 分）

### Step 1: 月曜 06:00 に reporter-agent が走る

`client-env/07-reports/YYYY-Wnn_weekly-report.md` が自動生成され、Slack #ai-pmo-reports に通知。

### Step 2: PMO 担当が 30 分でレビュー

ビューワーの「報告書」タブから最新報告書を開いて：

1. エグゼクティブサマリ（3 行）が事実と合っているか
2. 進捗パーセンテージに違和感がないか
3. 「重要トピック」セクションがクライアント目線で適切か
4. KPI トラッキングが正しいか

修正があれば直接 md を編集（Phase 2 で UI からも編集可能になる予定）。

### Step 3: PM が承認

- ビューワー上で内容確認
- 問題なければ Slack で「承認」スタンプ → 配信フロー起動

### Step 4: 週次定例で AI PMO ステータス確認

定例の冒頭 5 分で：
- 先週の AI 生成物の品質サマリ（誤り率、漏れ件数）
- watcher-agent の誤検知率
- 来週の AI PMO 運用上の改善点

### Step 5: 経営層向け Slack DM 配信

承認済みレポートを以下に配信：
- 経営スポンサーへ Slack DM（要約 + ビューワーリンク）
- メール（PDF 添付バージョン、ビューワーの「印刷/PDF」ボタンで生成）
- Confluence（アーカイブ用）

## 月次ルーチン（月初第1営業日、2 時間）

### Step 1: 月次経営報告の自動生成

reporter-agent が月初 06:00 に `client-env/07-reports/YYYY-MM_monthly-report.md` を生成。

### Step 2: PMO 室長 + PM のレビュー会（60 分）

- 月次レポートを全員で精査
- 数値の解釈（「なぜこの遅延が出たか」）を議論
- 経営層向けの **エグゼクティブメッセージ** を磨く（AI 下書きの 30% は修正）
- 翌月の重点課題を確定

### Step 3: PDF 化 + 経営報告会

- ビューワーの「印刷/PDF」で PDF 出力
- PowerPoint への変換が必要な場合は Excel データから別途整形（弊社側）
- 経営報告会で発表（PMO 室長 + Sr. コンサル）

### Step 4: KPI 集計 + Sr. コンサルとの振り返り

- ビューワーの KPI スナップショットを Excel に書き出し
- AI PMO の **目標 KPI 達成率** を集計
- Sr. コンサルと 60 分振り返り、翌月の調整事項を確定

## イレギュラー対応

### 課題が緊急発生したとき

1. ビューワーで該当課題を確認（または手動で `client-env/05-issues/issues.csv` に追加）
2. triager-agent が日次バッチで分類・優先度付け（緊急時は手動上書き）
3. High 優先度なら Slack #ai-pmo-alerts に即時投稿
4. PM が対応 → 解決後にステータスを Closed に

### AI 生成物が明らかにおかしい

1. Slack #ai-pmo-feedback チャンネルに「これおかしい」とポスト + スクショ
2. AI PMO エンジニア（弊社）が 1〜2 営業日でエージェント設定を修正
3. 再生成を実行

### ロールバック（AI を一時停止）

「AI PMO を一旦止めたい」となったら：

```bash
# 弊社運用チームが実行
gh workflow disable .github/workflows/weekly-report.yml --repo keitaurano-del/ai-pmo
gh workflow disable .github/workflows/daily-watcher.yml --repo keitaurano-del/ai-pmo
# n8n の cron ジョブも停止
```

過去の AI 生成物は GitHub 履歴に残るため失われない。

## 「使いこなしている」状態の指標

3 ヶ月運用した時点で以下が常態化していれば成功：

- [ ] 月次経営会議で AI PMO 生成資料がデフォルトで使われている
- [ ] PMO ジュニアでも週次レポートを 30 分以内に確認・承認できる
- [ ] 課題管理表に常時 30 件以上のアクティブ案件があり、滞留がない
- [ ] 議事録の確定リードタイムが平均 4 時間以内
- [ ] AI 生成物への修正率が 30% 以下
- [ ] PM 満足度が 4.0/5.0 以上

このうち 4 つ以上達成なら Phase 2（拡大）へ移行判定。

## よくある質問

### Q. AI が間違ったらどうする？

最終責任は PM / PMO にあり、AI 生成物は **すべて人間レビューを通る前提**。
ただし「全件チェックすると AI 化の意味がない」ので、サンプリング + 異常検知パターンで効率化する：
- 議事録: 全件レビュー（5〜10 分）
- 週次レポート: 全件レビュー（30 分）
- 課題自動起票: スポットチェック（10 件中 1 件）
- リスク予兆検知: 全件確認（5 件以下/週なら全部）

### Q. クライアント（M-Mart 等）に AI 利用を伝えるべき？

**伝える前提**。契約書・キックオフ資料で明示。
AI 生成物には必ず `Generated by: <agent>-agent (AI PMO)` フッターを付ける。
透明性が信頼を生む。

### Q. データ漏えいリスクは？

- API キーは Doppler / 1Password で集中管理、ローテーション 90 日
- AI への送信データは GitHub に残るが、リポは Private
- センシティブ情報（個人情報・他社機密）は事前に PII 検知でマスク
- 詳細は `docs/enterprise-considerations.md` 参照

### Q. AI 利用料が想定より膨らんだら？

- 月次でモニタ（`scripts/agents/run.ts` が usage を記録）
- Opus → Haiku への切替で 80% コスト削減（精度は下がる）
- ある程度の生成物は週次 → 月次に頻度ダウンで対応

### Q. クライアントの IT 統制で外部 API NG と言われた

- AWS Bedrock 経由で Claude を呼ぶ（VPC 内完結）
- Doppler を社内 Vault に置換
- Render の代わりに AWS ECS or EC2 へデプロイ

オンプレ運用の詳細は別途見積もり（追加 ¥3〜5M/年）。

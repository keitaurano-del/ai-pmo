# 法人で AI を使うときの留意点

AI PMO を企業に導入する際、PoC を本番に移す段階で必ず詰まる「ガバナンス・セキュリティ・コンプライアンス」観点を整理。

> 骨格のみ。各セクションは Keita のレビュー後に具体化する。

---

## 1. データ取扱

| 論点 | 確認事項 |
|----|-------|
| 入力データの機密度 | 顧客情報・人事情報・営業秘密が含まれるか |
| 学習利用の可否 | LLM ベンダーがプロンプトを学習に使う設定になっていないか |
| データ越境 | 海外リージョンに送ってよいか（個情法・GDPR） |
| データ保持期間 | ログ・キャッシュの保存期間と削除可否 |
| 匿名化・マスキング | 個人情報を投入する場合の処理方針 |

**ベストプラクティス**: 法人向けの「学習に使わない」契約条項（Bedrock / Azure OpenAI / Anthropic Enterprise 等）を必須化。

---

## 2. アクセス制御・認証

- 誰が AI PMO を使えるか（PM, PMO, 役員, 一般メンバー）
- 案件ごとのアクセス境界（プロジェクト A の人がプロジェクト B のデータを見られない）
- SSO（SAML / OIDC）連携の必須化
- 監査ログの保存・閲覧権限

---

## 3. 出力ガバナンス

- ハルシネーション対策（根拠提示・要事実確認のフロー）
- 自動アクション境界（読み取りのみ / 下書きまで / 自動発信OK）
- 重要判断は人間承認必須（特に金額・契約・要員）
- AI 出力に「これは AI による下書き」の明示

---

## 4. 契約・責任

| 論点 | 内容 |
|----|----|
| 責任分界点 | AI が誤った提案を出した場合の責任は誰か |
| SLA | 可用性・応答時間・エスカレ手順 |
| 知的財産 | 生成物の権利帰属 |
| 解約時のデータ返却・削除 | 出口設計 |

**典型的な落とし穴**: 顧客は「AI が出した答えに従って失敗したらどう責任を取るのか」を必ず聞いてくる。**「AI は下書き、最終判断は人間」を明示**して、責任は顧客側に留める設計が原則。

---

## 5. 業界別の追加要件

| 業界 | 追加考慮 |
|----|------|
| 金融 | FISC 安全対策基準、金融庁ガイドライン |
| 医療 | 個人情報保護法（医療情報）、3省2ガイドライン |
| 公共 | 政府クラウド（ガバメントクラウド）認定、調達ルール |
| 製造（防衛・輸出） | 外為法、技術情報の越境制限 |
| 上場企業全般 | 内部統制（J-SOX）、開示義務 |

---

## 6. 社内承認プロセス（顧客側で必要なステップ）

1. 情報システム部門のレビュー
2. セキュリティ部門のレビュー（脆弱性診断・PIA）
3. 法務部門のレビュー（契約条項）
4. リスク管理部門の承認
5. 役員会・経営会議の決裁

**コンサル側の支援**: 上記の各部門向け説明資料テンプレを `proposals/` に整備しておく。

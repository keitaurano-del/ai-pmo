## プロジェクト概要

### 目的

現行の受発注システム（2008 年稼働、レガシー COBOL ベース）を全面刷新し、以下を実現する：

1. **EC チャネルとの在庫リアルタイム連携**（現状はバッチ夜間更新）
2. **取引先 EDI 対応の最新化**（現行は flat-file SFTP、新規取引先は API/Web 注文を希望）
3. **配送計画の自動最適化**（現状は配送センター長の手作業）
4. **経営ダッシュボード**（売上・在庫・配送進捗をリアルタイム可視化）

### スコープ

#### 含むもの
- 商品マスタ管理（約 30,000 SKU、季節商品含む）
- 受注管理（B2B 800 社、B2C EC モール 5 社経由）
- 在庫管理（自社 3 配送センター、外部委託倉庫 5 拠点）
- 出荷指示・配送計画
- 取引先 EDI（Web API + flat-file 互換）
- 経営ダッシュボード
- AWS への基盤移行

#### 含まないもの
- 会計システム（既存パッケージ継続）
- 人事・給与システム（別案件）
- ECサイトのフロント側刷新（別ベンダー実施中）

## 既存システム環境

| 用途 | 現行 | 備考 |
|------|------|------|
| 受発注 | 自社開発 COBOL | 2008 年稼働、保守 SE 退職リスク |
| 在庫管理 | パッケージ A 社製 | カスタマイズ多数、追従困難 |
| EDI | flat-file SFTP | 取引先 200 社、夜間バッチ |
| 配送計画 | Excel + 手作業 | 配送センター長依存 |
| 経営報告 | PowerPoint 月次 | 情報システム部が手作業集計 |
| ファイル共有 | Box | 全社標準 |
| 業務コミュニケーション | Slack | 全社標準 |

## 想定リスク（弊社認識）

1. **COBOL 仕様の暗黙知**: ドキュメント不足、退職 OB ヒアリングが必要
2. **取引先 EDI 移行**: 200 社への切替案内 + テスト工程が膨大
3. **3 配送センター + 5 外部倉庫の同時切替**: ビッグバン or 段階移行の判断が必要
4. **新基幹に必要な Java エンジニア確保**: 全国的に逼迫

## 既に決定している事項

- 新基盤は AWS Tokyo Region に統一
- DB は Amazon Aurora (PostgreSQL 互換)
- フロントは React + TypeScript
- バックエンドは Java (Spring Boot) または Node.js のいずれか

## 用語

- **SKU**: Stock Keeping Unit（在庫管理単位、商品の最小管理単位）
- **EDI**: Electronic Data Interchange（取引先との電子データ交換）
- **B2B**: Business to Business（法人向け取引）
- **B2C**: Business to Consumer（一般消費者向け取引）

## 提案書に含めてほしい項目

- 体制・スケジュール・価格
- 技術選定の理由
- リスク管理計画
- パラレル稼働の具体手順
- カットオーバー判定基準
- 保守・運用フェーズの提案
